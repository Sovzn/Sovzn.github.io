双亲委派机制

什么是双亲委派机制

当某个类加载器需要加载某个.class文件时，它首先把这个任务委托给他的上级类加载器，递归这个操作，如果上级的类加载器没有加载，自己才会去加载这个类。

类加载器的类别

BootstrapClassLoader（启动类加载器）

c++编写，加载java核心库 java.*中的类,位置：\jre\lib\rt.jar，构造ExtClassLoader和AppClassLoader。由于引导类加载器涉及到虚拟机本地实现细节，开发者无法直接获取到启动类加载器的引用，所以不允许直接通过引用进行操作

ExtClassLoader （标准扩展类加载器）

java编写，加载扩展库中的类， 位置：\jre\lib\ext，开发者可以直接使用标准扩展类加载器。

AppClassLoader（系统类加载器）

java编写，加载程序所在的目录，如user.dir所在的位置的class，加载用户自己编写的类

CustomClassLoader（用户自定义类加载器）

java编写,用户自定义的类加载器,可加载指定路径的class文件

委派机制的流程图

双亲委派机制的作用

  • **防止重复加载同一个.class**。

    通过委托去向上面问一问，加载过了，就不用再加载一遍。保证数据安全。

  • 保证核心.class不能被篡改。

    通过委托方式，不会去篡改核心.class，即使篡改也不会去加载，即使加载也不会是同一个.class对象了。不同的加载器加载同一个.class也不是同一个Class对象。这样保证了Class执行安全。

实例测试：

类加载器的类别：

public class Car {
	public Car() {
		super();
	}

	public static void main(String[] args) {
	Car car1=new Car();
	Car car2=new Car();
	Car car3=new Car();
	
	System.out.println(car1.hashCode());
	System.out.println(car2.hashCode());
	System.out.println(car3.hashCode());
	Class aClass1=car1.getClass();
	Class aClass2=car2.getClass();
	Class aClass3=car3.getClass();//一个类只有一个Class对象，所以aClass1、aClass2、aClass3相同
	
	System.out.println(aClass1);
	System.out.println(aClass2);
	System.out.println(aClass3);
	
	ClassLoader classLoader1 =aClass1.getClassLoader();//获取类加载器
	//类加载器:AppClassLoader（系统类加载器或应用程序加载器）
	System.out.println(classLoader1); 
	//上级类加载器：ExtClassLoader（标准扩展类加载器）
	System.out.println(classLoader1.getParent()); //    位置：\jre\lib\ext
	//上上级加载器：BootstrapClassLoader（启动类加载器），无法直接获取到启动类加载器的引用，返回null
	System.out.println(classLoader1.getParent().getParent()); //   位置：\jre\lib\rt.jar
	}
}

双亲委派机制：

 package java.lang; //跟java官方包下的\jre\lib\rt.jar中的String.class同包同名

public class String {
	public String toString() {
		return "Hello";	
	}	
	public static void main(String[] args) {
		 String s =new String();
		 s.toString();
	}
}

执行结果：
    
  错误: 在类 java.lang.String 中找不到 main 方法, 请将 main 方法定义为:
  public static void main(String[] args)
  否则 JavaFX 应用程序类必须扩展javafx.application.Application
      
      /*
      加载流程：
      1.类加载器收到加载请求、
      2.将这个请求一直向上委托，直到启动类加载器
      3.启动类加载器检测是否能够加载这个类（在核心库rt.jar中去寻找有没有这个类），能加载就结束，使用当前的加载器，否则，抛出异常，通知子加载器进行加载。
      4.重复3完成加载   
      */

从执行结果可以看出，程序运行错误，说找不到main方法，但是我们在自定义的String类中已经定义了main方法。这是由于双亲委派机制：我们自定义的String类会从系统加载器开始向上委托：AppClassLoader→ExtClassLoader→BootstrapClassLoader，因为在启动类加载器BootstrapClassLoader中完成了加载（在\jre\lib\rt.jar下的存在java.lang.Sting.class），所以按照双亲委派机制，由于上级加载器已近完成了加载，所有系统加载器AppClassLoader无法再加载我们自定义的String类，因此才会报错。这也正好说明了双亲委派机制的第二个作用：保证核心.class不能被篡改，因为即使你想修改java原来的代码，比如自定义String类，它也不会去加载，它只会加载底层的String.class，所以自定义的String类根本不会生效。
